# v0.1 Baseline Notes

## Summary

v0.1 is an initial public documentation baseline for Engineering Agents. The repository is documentation-only and provides role-based engineering planning and review guidance, reusable operating aids, and public-safe workflow guidance for AI-assisted engineering work.

This draft is intended as public-facing baseline release note text that a maintainer can copy or adapt later. It is not a GitHub release, tag, changelog system, automation plan, certification statement, compliance claim, safety claim, production-readiness claim, or exhaustive catalog of every agent file.

## What v0.1 Includes

The current confirmed coverage includes documentation-only guidance for:

- Role-based engineering planning and review agents.
- Web frontend, web backend, and web design planning and review coverage.
- Native/desktop and mobile review guidance for platform, runtime, privacy, release, and boundary questions.
- AI application, agent workflow, AI vendor, and supply-chain review guidance.
- Product, data, experimentation, quality, security/privacy, infrastructure, governance, review, and Codex documentation workflow guidance.
- Reusable templates for Codex docs PR prompts, PR bodies, routing notes, merge-readiness reviews, and evidence/confidence review.
- Checklists for PR review, merge readiness, public OSS hygiene, AI session/prompt hygiene, tool routing, agent pack maintenance, cost-aware workflow, and evidence/confidence review.
- Onboarding docs, common workflows, maintainer guidance, taxonomy, roadmap, tool routing, cost-aware workflow guidance, language/locale guidance, and review precision guidance.

Agents are role-based planning and review guides. Templates and checklists are operating aids, not agents. Onboarding and usage docs explain how to use the repository; they are not agent domains.

Cost-aware guidance is about choosing the smallest effective scope, context, authority, validation, and review effort for the task. It is not provider pricing advice, billing analysis, or a promise of exact cost savings.

Review precision guidance supports evidence/confidence calibration, scoped review language, and clearer separation of assumptions, limitations, blockers, non-blocking improvements, and owner decisions. It is not proof of global correctness, safety, compliance, privacy, security, or production readiness.

## How to Start

A practical starting path is:

1. Read `README.md` for the repository overview and current scope.
2. Read `docs/getting-started.md` for purpose, scope, and recommended first steps.
3. Use `docs/agent-taxonomy.md` to find the narrowest relevant agent area.
4. Use `docs/using-agents.md` and `docs/common-workflows.md` to combine agents with templates and checklists.
5. Use `docs/tool-routing.md` and `checklists/tool-routing-decision-checklist.md` before choosing GPT, Codex, Claude Code, local/manual work, or human-only review.
6. Use `docs/cost-aware-workflows.md` and `checklists/cost-aware-workflow-checklist.md` to keep context, validation, review, and changed-file scope proportional to the task.
7. Use `docs/review-precision.md`, `templates/evidence-confidence-review-template.md`, and `checklists/evidence-confidence-review-checklist.md` when review output needs calibrated evidence, confidence, limitations, and owner-decision language.

For repository changes, keep work small, branch-based, PR-reviewed, validation-backed, documentation-only, and public-safe.

## Public OSS Hygiene

Repository-facing content should remain public-safe. Do not include private AI session links, Codex task URLs, Claude session URLs, ChatGPT shared links, private prompts, hidden prompts, production prompts, AI session logs, AI transcripts, model traces, tool logs, credentials, secrets, customer data, private dashboards, private roadmap details, legal analysis, compliance analysis, procurement notes, or internal-only notes.

Also avoid local paths, generated-by signatures, private tool metadata, tokens, API keys, participant data, personal data, sensitive data, user content, support tickets, logs, crash reports, stack traces, analytics exports, dashboard exports, private review evidence, private deployment or preview URLs, private billing URLs, private usage dashboards, private governance notes, private maintainer notes, private provider dashboards, vendor details, contract terms, security questionnaires, and unreleased private business context.

Use `checklists/public-oss-hygiene-checklist.md` and `checklists/ai-session-prompt-hygiene-checklist.md` before publishing docs, templates, checklists, PR bodies, or review comments.

## What This Is Not

The v0.1 baseline is not:

- a complete, final, certified, compliant, production-ready, fully safe, or exhaustive system;
- a GitHub release, tag, changelog automation, release workflow, or marketing campaign;
- application code, implementation code, scripts, generated files, package files, lockfiles, dependencies, build tooling, CI workflows, provider setup, screenshots, diagrams, assets, billing dashboards, usage reports, model logs, tool logs, legal templates, compliance documents, certification artifacts, or `.github` issue/PR templates;
- proof that any external product, workflow, PR, model, provider, vendor, dependency, security posture, privacy posture, legal position, or compliance posture is correct or ready;
- a pricing guide, provider comparison, billing report, usage dashboard, or claim of exact cost savings;
- a generated catalog of every agent file.

Human owner decisions remain required for merge decisions, releases, private data, legal/procurement questions, security-sensitive findings, destructive actions, external communications, and ambiguous authority.
